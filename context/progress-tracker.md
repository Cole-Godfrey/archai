## Architecture Decisions

- shadcn/ui is configured with generated primitives in `components/ui/*`; project-specific theming lives in `app/globals.css`.
- Clerk public auth paths are derived from the standard `NEXT_PUBLIC_CLERK_SIGN_IN_URL` and `NEXT_PUBLIC_CLERK_SIGN_UP_URL` env vars, with `/sign-in` and `/sign-up` as local route fallbacks.
- Project API routes perform their own Clerk auth checks so unauthenticated API requests can return JSON `401` responses instead of `auth.protect()` API `404` responses.
- Project workspace access checks are centralized in `lib/project-access.ts`; `/editor/[roomId]` treats room IDs as project IDs.
- Editor pages perform server-component auth redirects to `/sign-in`; proxy-level Clerk protection is skipped for `/editor` routes so page-level access handling can render `AccessDenied` for missing or unauthorized projects.
- Project collaborator records remain email-only in PostgreSQL; share-dialog display names and avatars are enriched at request time from Clerk Backend API with email-only fallback.
- Share-dialog access lists include the project owner derived from `Project.ownerId`; owners are not stored as collaborator rows.
- Share-dialog API payloads do not expose `Project.ownerId` as a contact field; unresolved owners render with a generic owner label.
- Share-dialog owner invite prevention compares invite emails against every normalized email address on the owner's Clerk account, with the signed-in primary email retained as a fallback.
- Liveblocks auth uses private project rooms and issues room-scoped access tokens only after Clerk authentication and project membership verification.
- Liveblocks user metadata uses Clerk display data plus a deterministic Archai cursor color derived from the Clerk user ID.
- Canvas autosave stores private Vercel Blob JSON at `canvas/{projectId}.json`, stores the returned URL in `Project.canvasJsonPath`, and loads saved snapshots only when the Liveblocks room has no active nodes or edges.
- Trigger.dev is the durable background-task layer: `@trigger.dev/sdk` (dependency) and `@trigger.dev/build` (dev dependency) at v4, configured in `trigger.config.ts` with `dirs: ["./trigger"]`, dev-disabled retries, and a 3600s max duration. Tasks live in the root-level `trigger/` directory per the architecture system boundaries.
- `trigger.config.ts` loads `.env.local`/`.env` via dotenv at the top (the Trigger.dev CLI evaluates the config through jiti *before* hydrating env), then reads the project ref from `TRIGGER_PROJECT_REF` and throws if unset; the SDK reads `TRIGGER_SECRET_KEY` at runtime. Both live in `.env.local` (gitignored, not committed) and must be set before the dev server or task triggering will work.
- Design generation backend wiring (feature 22): `POST /api/ai/design` authenticates, validates `{ prompt, roomId, projectId }`, enforces project ownership via `getProjectAccessForIdentity`, triggers the `design-agent` task with `tasks.trigger<typeof designAgentTask>` (imported with `import type` so only the task-id string couples to the route), records a `TaskRun` row, and returns `{ runId }` with `202 Accepted`. `POST /api/ai/design/token` re-authenticates, looks up the `TaskRun` by `runId`, requires `taskRun.userId === user.userId` (returns `403` otherwise, not leaking existence), and issues a read-only run-scoped token via `auth.createPublicToken({ scopes: { read: { runs: [runId] } }, expirationTime: "1h" })`. Request parsing lives in `lib/ai-design.ts`; response/auth helpers are reused from `lib/project-api.ts`.
- `TaskRun` is a standalone Prisma model in `prisma/models/task-run.prisma` (no `@id`, mirroring `ProjectCollaborator`): `runId` is `@unique` (which provides the required runId index and serves as the lookup identifier for `findUnique`), plus a compound `@@index([userId, projectId])`.
- Design agent logic (feature 23): the `design-agent` task interprets `{ prompt, roomId }` with Gemini and writes a structured design onto the shared canvas. Gemini access uses `@ai-sdk/google` `createGoogleGenerativeAI({ apiKey: GOOGLE_AI_API_KEY })` with `ai`'s `generateObject` and a zod schema; the model defaults to `gemini-3.5-flash` and any `GOOGLE_AI_MODEL` override must be a `gemini-3*` model id. The model emits a `summary` plus a discriminated-union list of `actions` (`add_node`/`update_node`/`move_node`/`resize_node`/`delete_node`/`add_edge`/`delete_edge`); shape/color are constrained to `NODE_SHAPES`/`NODE_COLORS` via zod enums. New-node placement is now model-directed: the prompt includes existing node positions, sizes, centers, edges, and the requester's viewport center; `add_node` schema validation requires finite `x`/`y` coordinates and a color; `prepareDesignPlan` trusts those coordinates, generates collision-free ids, clamps optional sizes, resolves edge/edit references, and expands generated-node spacing for added edges before applying the plan. Edge-label spacing targets 300px center-to-center normally and 400px for labels at least 18 characters, moving only generated nodes while preserving existing-node move actions. `add_edge` schema validation requires a non-empty descriptive label, and there is no generic edge-label fallback. Gemini generation is bounded with a 45s `AbortController` timeout so invalid prompts or provider stalls fail quickly before any canvas mutation. Google provider structured outputs are disabled for this call because Google can stall on discriminated unions; the worker uses Gemini JSON mode, Gemini 3 high thinking, and strict local zod validation instead. When Gemini returns invalid structured output, the worker now revalidates the raw response locally and logs structured diagnostics to Trigger.dev: schema issue paths/messages, raw Gemini response text, finish reason, model id, usage, and the original AI SDK cause.
- Canvas writes go through the existing collaborative flow's backend counterpart: `mutateFlow` from `@liveblocks/react-flow/node` (same default `flow` storage key as the editor's `useLiveblocksFlow`), driven by the `@liveblocks/node` client. The task reads current canvas state via a read-only `mutateFlow` pass (so the model can extend, not just overwrite), then applies all changes in a single `mutateFlow` write. Deleting a node also removes its connected edges so no dangling edges remain.
- AI presence + status are published with the `@liveblocks/node` client's `setPresence` (ephemeral, no WebSocket) under a fixed agent identity (`AI_AGENT_USER_ID`/`AI_AGENT_NAME`/teal `AI_AGENT_COLOR` in `types/ai-design.ts`). The agent's cursor/avatar reuse the existing `LiveCursors`/`ParticipantAvatarGroup` rendering; the global Liveblocks `Presence` gained an `aiActivity: AiAgentActivity | null` field (a type alias, not an interface, so it satisfies Liveblocks' `Json` validation) carrying `{ phase, message }`. A new `AiAgentStatus` canvas overlay renders that activity for all participants. Active phases use a 60s presence TTL (refreshed each step and slightly longer than the 45s generation timeout); terminal phases (`complete`/`error`) use a 6s TTL so the agent clears itself when finished.
- The `design-agent` task sets `retry: { maxAttempts: 1 }`: generation runs before any canvas write, so a failure leaves the canvas untouched, and a retry would otherwise append a second, different design to a shared room. Errors publish an `error` status and re-throw without mutating the canvas.
- AI presence/status consumption (feature 24): a single `LiveblocksProvider` + `RoomProvider` now lives in `EditorWorkspaceShell` (wrapping `BaseCanvas`, `AISidebar`, and `ProjectSidebar`) instead of inside `BaseCanvas`, so the sidebar and canvas share one room connection — the Liveblocks-documented "one room provider, multiple regions each with their own suspense/error boundary" pattern. `BaseCanvas` no longer owns the providers (or takes `roomId`); it keeps its own `CanvasErrorBoundary` + `ClientSideSuspense` + `LiveblocksConnectionFallback`, and `initialPresence` moved up to the shell's `RoomProvider`.
- The shared AI status channel is the Liveblocks **Feeds API** (`@liveblocks/react` `useFeedMessages`), not a parallel realtime state. The feed id is `AI_STATUS_FEED_ID = "ai-status-feed"` (`types/tasks.ts`), generic enough for both design and spec generation. The sidebar subscribes with the non-suspense hook and renders only the most recent validated message, so a missing/empty feed (no writer yet) leaves the rest of the sidebar usable. The message payload is typed globally via `Liveblocks["FeedMessageData"] = AiStatusFeedMessageData` (a `{ text?: string }` type alias, not an interface, so it stays `Json`-assignable) and is still validated at the boundary by `parseAiStatusFeedMessageData` before display.
- Liveblocks `feeds:write` is not part of `session.FULL_ACCESS` (which is `["room:write"]`, covering only room storage and comments), so `/api/liveblocks-auth` now grants `[...session.FULL_ACCESS, "feeds:write"]`. Authorized project members (already verified before token issuance) can therefore read and write the room's `ai-status-feed`.
- Liveblocks feeds must be **created before they can be written** — posting a message to a feed that was never created fails server-side (verified: a `createFeedMessage` to a non-existent feed returns HTTP 500, while the same call after `createFeed` succeeds). The client hooks only read/write (`useFeedMessages`/`useCreateFeedMessage`) and never create a feed, so `/api/liveblocks-auth` now calls `ensureRoomFeeds(roomId)` (in `lib/liveblocks.ts`) right after `getOrCreateRoom`, provisioning both `ai-status-feed` and `ai-chat`. It is idempotent (lists existing feeds via `getFeeds`, creates only the missing ones) and best-effort (never throws, so it can't block authorization); it runs once per room connection, not per message. This self-heals every existing room on its next connect.
- Sidebar chat feed (feature 25): collaborative room chat uses a second, independent Liveblocks **Feeds API** channel, `AI_CHAT_FEED_ID = "ai-chat"` (`types/tasks.ts`), kept strictly separate from `ai-status-feed` — status and chat never share a feed. The chat payload (`{ sender, role: "user" | "assistant", content, timestamp }`) is defined by a zod schema in `types/tasks.ts` (`aiChatFeedMessageDataSchema`) and validated at the boundary by `parseAiChatFeedMessageData` before any message renders; `role` reserves `assistant` for a future AI reply flow but only `user` is produced today. The global Liveblocks `FeedMessageData` is now a union `AiStatusFeedMessageData | AiChatFeedMessageData`, with each reader validating via its own parser. `hooks/use-ai-chat-feed.ts` subscribes with the non-suspense `useFeedMessages` (tolerating a missing/empty feed), orders entries by authored `timestamp`, flags own messages via `useSelf` name match, and sends through `useCreateFeedMessage` — clearing the input on success and surfacing a small inline error on failure. The sidebar's existing input/send button drive it; a local submitting/run guard prevents duplicate sends from the submitting client. No AI replies and no backend task triggering were added in feature 25.
- "AI is working / generation active" presence is derived from the existing agent **presence** (`aiActivity` active phase via the shared `isActiveAiPhase` helper in `types/ai-design.ts`), not from the feed. Canvas overlays and live cursor badges remain room-wide. The sidebar reads the latest `ai-status-feed` text, but its input/chips/send button are gated by the submitting client's local run/submission state instead of room-wide presence so stale or missed terminal presence cannot lock chat.
- Design agent frontend (feature 26): the AI Architect tab in `components/editor/ai-sidebar.tsx` now drives generation. On submit it pushes the participant's `user` message to `ai-chat`, then triggers the run via `POST /api/ai/design` and resolves a run-scoped token via `POST /api/ai/design/token`, tracking it with `@trigger.dev/react-hooks` `useRealtimeRun` plus a `useRun` polling fallback (`accessToken` gated on a local `{ runId, token }` state, so it never fetches without a token before a run exists). The spec described a single `{ runId, publicToken }` response, but feature 26's scope forbids backend changes and the two endpoints already existed (features 22–23, the token split is a deliberate re-auth/ownership check), so the frontend uses the existing two-call flow to satisfy the same intent. `roomId`/`projectId` are threaded from `EditorWorkspaceShell` (both equal `currentProject.id`); `POST /api/ai/design` requires both even though the spec listed only `roomId`.
- The submitter's sidebar watches terminal run state (`finishedAt`, `isCompleted`, or terminal Trigger.dev status) and posts a single closing `assistant` message to `ai-chat` (the run's `output.summary`, or a generic fallback; failures/cancellations/timeouts post a "generation failed" notice). A `handledRunIdRef` guard ensures exactly one closing message per run, and only the submitting client (the only one holding `runId`+token) tracks the run — so the shared feed gets one message that every client sees. Start-time backend errors (`/api/ai/design` or token failures) likewise surface as an `assistant` message; the participant's own chat-send failures keep the existing inline `sendError`. `sendAssistantMessage` was added to `hooks/use-ai-chat-feed.ts` (posts `role: "assistant"` under `AI_AGENT_NAME`) so all `ai-chat` writes stay in that one hook.
- Sidebar chat bubbles are styled by `role`: `user` → teal `bg-ai` (`#38D7BD`) with `text-primary-foreground` for contrast; `assistant`/AI → `bg-elevated` with `text-copy-primary` (dark surface, light text). Own-vs-other is still conveyed by left/right alignment (`isOwn`). The Send button is teal (`bg-ai`) when enabled, dimmed when disabled, and shows a spinner while the local submit/run is busy. The status strip (dark `bg-elevated` base + animated teal `bg-ai-text` dot) renders only while the local tracked run is active, reading its text from `ai-status-feed`. Canvas/edge/presence updates are not synced manually — they keep arriving through the editor's existing `useLiveblocksFlow`, which the background agent writes to via `mutateFlow`.
- The Prisma client (`lib/prisma.ts`) wraps every model operation in a `$extends({ query: { $allOperations } })` retry. Free-tier Prisma Postgres (the `pooled.db.prisma.io` endpoint) auto-pauses after inactivity and returns "Failed to connect to upstream database" on the first request while it wakes; the retry catches that (and other connection-*establishment* failures: `PrismaClientInitializationError`, `P1001`/`P1002`, `ECONNREFUSED`/`ETIMEDOUT`/`ENOTFOUND`/`EAI_AGAIN`, walking the `error.cause` chain) and retries up to 4 attempts with 250/500/1000ms exponential backoff before throwing. Scoped to connection-establishment errors only, so a retry can never duplicate a write (the query never executed). The singleton type is `ReturnType<typeof createPrismaClient>`; no call site uses `$transaction`/`$queryRaw`/raw APIs, so the extension is transparent.
